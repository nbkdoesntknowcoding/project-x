How Mnema syncs

Every edit is captured by Yjs and broadcast through Hocuspocus to all connected clients within 50ms. Persistence flushes to Postgres on a 3-second debounce, batching rapid edits without losing intermediate state.

Claude reads from this same surface through MCP. Because there is no upload step, the doc Claude reads at 3:14pm is the doc your teammate edited at 3:13pm.

MCP read path

Claude calls get_doc on the MCP server, which returns the markdown of the doc as it exists right now — no caching, no staleness window. The server reads the latest Yjs snapshot from Postgres and serialises it on the wire.

Hybrid search

Search uses a hybrid scorer: BM25 over titles plus a cosine match over chunk embeddings. The MCP tool search_workspace returns top-k results with surrounding context.

MCP CALL · search_workspace 
// returns top-5 results with citations
POST /mcp/tools/search_workspace
{
  "query": "rate limiting",
  "k": 5,
  "include_chunks": true
}

Why this matters

Static doc uploads break the moment you edit them. Claude is reading from a snapshot taken whenever you last clicked sync. With Mnema, sync is the read path — there is no upload step to forget.

  • Edits propagate in under 50ms
  • Claude reads the live state, not a copy
  • No re-indexing, no re-uploading
  • Audit log shows every MCP call against your workspace
